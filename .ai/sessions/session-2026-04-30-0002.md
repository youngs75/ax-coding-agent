### 3. 근본 원인 — `_extract_todos` 가 None 반환

[`coding_agent/tools/todo_tool.py:220`](coding_agent/tools/todo_tool.py#L220):
```python
return render_todo_summary(items)  # ← 사람-읽기-용 string 반환
```

[`coding_agent/web/sse_emitter.py:218-251`](coding_agent/web/sse_emitter.py#L218)
의 `_extract_todos` 동작:
- `list[dict]` 면 변환 ✓
- `str` 이면 **`json.loads` 시도** → 실패 시 `None` 반환

`render_todo_summary` 가 반환하는 `Todos: 2 total — pending=1, ...` 형태는
JSON 이 아님 → 파싱 실패 → **`None` 반환 → emit skip → wire 에 0건**.

**v22.2 (2026-04-26 ledger SubAgent 폐기 → planner 가 write_todos 직접
호출) 부터 잠재된 결함**. CLI 모드는 별도 `on_change` callback 으로 패널을
갱신해서 영향 없었고 (CLI 가 `TodoStore` 에 직접 접근), web 모드는
`_extract_todos` 경로만 사용 → 모든 todo update 가 silently dropped.

### 4. 직전 핸드오프 §1 의 "todo.change ✓" 가 사실 잘못된 추론이었음

0001 의 §1 "fix B 진단 logging 캡처 분석" 표에 적힌 "kind=on_tool_end
name=write_todos depth=1 → write_todos priority (SubAgent 억제 전) →
`orchestrator.todo.change` ✓" 는 *코드 분기 도달* 까지만 확인했고 *실제 wire
도달* 은 미검증. chat_common.js parser 결함이 동시에 있어서 사용자 화면에서
"안 보임" 으로 가려졌고, parser fix 후에야 결함이 단독 노출됨.

→ **결함 A 가 결함 B 를 마스킹한 두 번째 사례** (어제는 ALB timeout 이
write_todos SubAgent 우회 fix 효과를 마스킹, 오늘은 chat_common.js parser
결함이 todo.change emit 누락을 마스킹).

### 5. Fix — store reference 주입 + snapshot 우선 사용 (`8ff5ccc`)

[`coding_agent/web/sse_emitter.py`](coding_agent/web/sse_emitter.py) 변경:

- `stream_agent_events` 가 `AgentLoop.get_todo_store()` 호출해서 store
  reference 획득 → `map_state["todo_store"]` 에 저장
- 새 helper `_todos_from_store(store)` — store 의 `list_items()` 호출해서
  source-of-truth 를 직접 surface (TodoItem 의 id/content/status 추출)
- write_todos / update_todo end 분기는 store snapshot 우선, store 없으면
  (단위 테스트 경로) 기존 `_extract_todos(output)` fallback

```python
todos = _todos_from_store(state.get("todo_store"))
if todos is None:
    todos = _extract_todos(data.get("output", ""))
if todos is not None:
    return sse("orchestrator.todo.change", {"todos": todos})
```

→ tool 의 반환 형식 (string vs list) 변경에 견고. `render_todo_summary` 가
계속 사람-읽기-용 string 을 반환해도 무관 (LLM 측 next-turn context 용도).

## 중요 참고사항

### `TodoStore` 가 source-of-truth, tool output 은 LLM 용 summary

이번 결함의 교훈 — **tool 의 반환 값과 backend state 는 다른 audience 를
위한 다른 representation**. `write_todos` 의 `_run` 이 string 을 반환하는
이유는 LLM 이 다음 turn 에서 todos 를 *읽어서* progress 추론하기 위함
([todo_tool.py:172-189](coding_agent/tools/todo_tool.py#L172) 의
`render_todo_summary`).

→ 새 tool 추가 시 LLM 용 tool output 과 wire-format 용 state 둘 다
고려해야 함. wire 에서 source-of-truth 가 필요하면 store reference 를 주입.

### 결함 A 가 결함 B 를 마스킹한 두 번째 사례 — 진단 원칙 강화

| 사례 | 결함 A (마스커) | 결함 B (피마스킹) |
|---|---|---|
| 2026-04-29 | ALB 60s timeout | write_todos SubAgent 우회 fix |
| 2026-04-30 (오늘) | chat_common.js SSE parser | sse_emitter todo.change emit 누락 |

**원칙**: 어떤 fix 의 효과를 평가할 때
1. graph/stream 이 끝까지 완주 가능한 환경에서
2. 진단 logging 으로 실제 이벤트 흐름을 확인한 *후*
3. 해당 fix 의 효과 판단

이 순서를 어기면 *허상의 결함 추가 fix* 위험. 본 세션은 §3-4 에서 그 위험을
피하기 위해 ax 컨테이너 로그 → wire response 순서 확인 후 코드 검토.
