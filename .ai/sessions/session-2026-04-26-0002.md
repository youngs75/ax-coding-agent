### config.py
```python
_ZAI_MODELS = ModelTier(

    strong="glm-5.1",        # coder/orchestrator
```

### models.py
- `provider == "zai"` 분기 — `disable_streaming=True` 강제 (GLM 도 reasoning_content 모델)
- `_get_langfuse_callbacks()` lazy singleton — `LANGFUSE_PUBLIC_KEY/SECRET_KEY` 있으면 `from langfuse.langchain import CallbackHandler` 부착
- ChatAnthropic + ChatOpenAI(direct) 양쪽에 `callbacks=` 전달

- litellm 분기에 GLM/deepseek/qwen3-max/reasoner 패턴 매칭 시 `disable_streaming=True` 자동 적용 (v16 stream_chunk_timeout fix)

## v16/v17 실패 분석 (재현 방지용 박제)

### v16 (litellm + glm-5.1-coding 1차)
- 증상: planner SubAgent 가 144.7s 만에 `langchain_openai.stream_chunk_timeout fired`
- 원인: GLM 도 thinking 모델 (reasoning_content 반환) → langchain-openai 1.2.x 의 chunk 파서가 무한 대기
- Fix: litellm 분기에 GLM 패턴 매칭 → `disable_streaming=True`

### v17 (litellm + glm-5.1-coding 2차, streaming fix 적용 후)
- 증상: planner #2 가 PRD 4 청크 read 후 *8분 침묵* → watchdog_timeout(540s) → orchestrator retry → z.ai 429 cascade → `consecutive_errors_3` safe stop
- 원인 1: `glm-5.1-coding` 은 thinking 분량 과다. 단일 호출에서 TASK_BREAKDOWN.md (수천 토큰 + reasoning_content) 생성에 hang
- 원인 2: litellm 의 `Available Model Group Fallbacks=None` — 429 시 fallback 없음
- 원인 3: z.ai 무료 tier rate limit 짧음 (3 retry × 21s 모두 429)
- Fix: glm-5.1-coding → glm-5.1 다운그레이드 + provider 자체를 z.ai 직접 호출로 (litellm 우회)

## Langfuse 트레이싱 — 두 층 분리 박제

| 층 | 누가 발화 | 모드 |
|---|---|---|
| Orchestration (role 호출, pipeline step, tool call) | `LangfuseForwardObserver` 직접 SDK | 모든 모드 |
| LLM generation (prompt, tokens, cost) | litellm proxy / langfuse-langchain CallbackHandler | proxy 모드는 자동 / direct 모드는 신규 부착 |

direct provider 모드에서 cost tracking 손실 우려는 이번 PR 로 해소.
