## 중요 참고사항

### Nudge guard 설계 결정
- **왜 `_MAX_PENDING_NUDGES=3`?** 1 은 너무 보수적 (모델이 한 번 실수하면
  포기), 5+ 는 "모델이 결국 안 따라오는데 계속 찌르기" 낭비. 3 이면
  "한 번 실수, 한 번 더 실수, 마지막 경고" 의 자연스러운 재시도.
- **왜 HumanMessage?** SystemMessage 를 또 주입하면 기존 system prompt 와
  양립 어색. HumanMessage 로 "사용자가 지적한 것처럼" 넣으면 모델이
  자연스레 대응하도록.
- **왜 pure 함수 분리?** `_should_nudge_pending` / `_build_pending_nudge_message`
  를 모듈 레벨로 꺼내서 closure 밖에서 테스트 가능. 그래프 전체를 세우지
  않고 판정·렌더 로직만 스냅샷.

### Langfuse trace 활용
cloud.langfuse.com API 직접 호출 가능:
```bash
source .env && curl -s -u "$LANGFUSE_PUBLIC_KEY:$LANGFUSE_SECRET_KEY" \
  "https://cloud.langfuse.com/api/public/traces?fromTimestamp=...&toTimestamp=..."
```
트레이스 id 잡은 뒤 `/api/public/traces/{id}` 로 input messages / output
전체 json 조회. 디버깅 효율 매우 좋음.

### 네임스페이스 분리 원칙 (박제)
- `orchestrator.*` = Orchestrator 자신의 행동 (run/pipeline/role.invoke/
  hitl/memory/resilience.escalate)
- `role.*` = 현재 invoke 된 role 안에서 ToolInvocationEngine 이 내는
  tool-level 이벤트. 어떤 role 인지는 enclosing `orchestrator.role.invoke`
  span 으로 식별 — 개별 tool-call 이벤트에 role 필드를 달지 않는 이유는
  `ToolInvocationEngine` 이 role-agnostic 이라는 계약 때문 (라이브러리
  docstring 참조).
