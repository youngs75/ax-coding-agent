### 3. wire 순서 분석 — `구현 → 검증 → 검증완료 → 구현완료`

`_auto_verify_chain` 이 coder 의 `_run_wrapped` *안에서* wrapping:
1. `on_tool_start name=task` → `구현 (coder)` 시작
2. coder `_run()` → COMPLETED 마커 반환
3. auto-chain 진입 → verifier `start` listener emit
4. verifier 실행 → success
5. verifier `end` listener emit
6. `_auto_verify_chain` 반환 → `_run_wrapped` 반환
7. `on_tool_end name=task` → `구현 (coder)` 완료

→ `구현 완료` 가 `검증 완료` *후에* 발행되는 게 **정확한 실행 흐름**. UX 관점에선
살짝 어색할 수 있으나 cosmetic 차원이라 처방 큐에만 등록 (선택적).

### "안 보이는 것" 은 "안 쓰는 것" 이 아니다 — 진단 원칙

오늘 0004 사이클에서 사용자가 "verifier/fixer 는 왜 안 쓰지?" 라고 보고 →
실은 호출됐지만 wire 에 안 보였던 것. **사용자가 시스템 행동에 의문 가질 때
*그 행동이 wire/UI 에 보이는지* 항상 먼저 확인**. 백엔드 로그 + 산출물 + wire
response 교차 검증.

→ **모든 SubAgent 협응이 chat UI 에 보이는 것이 시스템 신뢰성의 핵심** (사용자
명시 표현). 본 세션이 그 목표 달성.

### v22 자체 회귀 vs portal 통합 layer 결함

v25 백엔드 8/8 PASS 였고, 4 ~ 7 day 주의 회귀는 모두 *portal 통합 layer*
(apt-web ↔ ax SSE wire format / write_todos emit / auto_advance / auto_verify
chain) 였음. v22 자체는 견고. v22.4 처방 큐는 portal 배포 트랙 시작 시점에
보류됐던 main work 로, portal e2e *완전 정상* 도달했으니 복귀 가능.

### ContextVar listener pattern 의 가치 재확인

`task_tool` (coding-domain) ↔ `sse_emitter` (web) 직접 import 의존 없이
cross-layer 통신. CLI / headless 모드는 listener 등록 안 함 → default `None`,
emit 비용 0. portal 환경에서만 stream 별 listener 등록 + 격리 (ContextVar
token reset 으로 leak 방지). 향후 다른 white-box 호출 (critic / sufficiency
loop auto-prompt) 도 같은 패턴으로 surface 가능.
