## 설계 의사결정

### 1) nudge guard: cumulative → progress-based
구조: `unfinished` 가 감소하면 stuck 카운터 0 리셋. stuck 만 누적되면 3회에
terminate. 생산적 agent 가 진전 중 잠시 쉬어도 터미네이트 안 됨.

**Why**: v1 에서 생산적인 coder(25 task 완료 후 계속 진전 중)가 cumulative
카운터로 silent-terminate 되는 걸 목격. 진정한 stuck 판별은 "실제 진전
유무" 가 핵심 시그널.

### 2) Decomposition HITL gate
`_GATED_ROLES={"coder","verifier","fixer","reviewer"}` delegation 직전,
`pending == total and not decomposition_confirmed` 이면 ToolMessage 로 차단.
SYSTEM_PROMPT 가 "planner 가 분해 수준을 사용자에게 물어야 함" 을 명시.

**Implicit confirm**: planner 가 이미 `ask_user_question` 을 호출한 기록을
감지하면 게이트 자동 해제. v3 의 중복 질문 패턴 차단.

**Why**: v2 에서 planner 가 과도하게 coarse 하게 분해(8 task)한 뒤 coder 가
한 번에 너무 많은 기능을 완료 선언. v5 에서 반대 방향(72개 과도 세분)도
같은 게이트로 자연 catch 됨.

### 3) ProgressWatchdog (라이브러리 계층)
wall-clock `asyncio.wait_for` 는 "진전은 있지만 단일 호출이 오래 걸리는"
케이스를 stuck 으로 오판. 대체: 도구 호출 성공마다 deadline 을 extend_s
연장하되 max_total_s 상한. 중복 args_hash 는 연장 효과 없음
(ProgressGuard 와 동일한 중복 탐지).

**Why 라이브러리로**: ax 계층에서만 풀면 apt-legal 등 다른 consumer 가
동일 문제를 재해결해야 함. harness 5책임 중 **Safety** 의 일부.

**검증**: v5 최장 coder 호출 421.2s × 48 iterations. base=240s 였던 상태면
세 번은 timeout 났을 길이. 새 도구 호출마다 +60s 연장되며 전부 통과.
