### 이번 세션의 교훈
1. **"규칙 추가 → 품질 향상"의 함정**: Iteration 1~2에서 프롬프트에 ABSOLUTE RULE/Scope lock/Phase 상세를 계속 쌓았지만, LLM이 오히려 각 규칙에 할당되는 attention이 분산되어 "minimum-turkey"로 수렴. **약한 모델일수록 프롬프트는 짧아야 한다**. Iteration 3에서 방향 전환 후 품질이 오히려 회복됨.
2. **도구 레벨 강제 > 프롬프트 지시**: `write_file`의 SPEC 경로 거부, `execute`의 watch 명령 reject, `spec_tool`의 REJECTED 메시지 등은 LLM이 무시할 수 없는 신호. "LLM에게 설명" 대신 "LLM이 실패하고 가이드 메시지에서 학습"으로 구조 전환.
3. **실측으로 가설 검증**: 3 라운드(v4/v5/v6)를 연속으로 돌리고 매번 Langfuse trace로 확인. "Phase 1/2 collapse"는 trace로 확인 전까지 의심만 했고, `write_todos` 필요성도 v6 E2E에서 orchestrator가 tool call 없이 종료하는 것을 보고 구조적으로 증명.
4. **도구 경계는 프롬프트보다 강하다**: Option A에서 fixer 프롬프트를 "do not run tests"로 바꿔도 LLM이 지킬 가능성은 있지만, **default_tools에서 `execute`를 제거하면 LLM 자체가 호출 불가능**. 이후 v6 E2E에서 fixer의 execute 호출 0건으로 확인.
5. **레퍼런스 벤치마킹의 가치**: Claude Code의 watch command list, CI=1 env injection, stall watchdog 패턴을 3개 레포 조사에서 학습. "우리가 빠뜨린 것"을 체계적으로 찾는 효과적 방법.
