### 6. LangGraph interrupt replay-safety 근본 수정
Qwen3 전환 후 HITL 이 **1 라운드 밀림** 현상. 사용자가 "항상 마지막 출력이
첫 단계 첫 줄" 이라는 패턴도 보고. 조사 결과:

**원인 1 (HITL 밀림)** — LangGraph 공식 pitfall. `task_tool._run_task` 가
`orchestrator.invoke_role` (LLM 호출, 비결정) 이후 `interrupt()` 를 호출하는
구조. resume 시 **tool 전체가 replay** 되므로 role.invoke 가 새로운 pending 을
만들고, interrupt 는 직전 answer 를 반환 → pending(신) × answer(구) mismatch.

수정: `InjectedToolCallId` 로 tool call 식별자 주입, module-level
`_TOOL_CALL_CACHE[tool_call_id][iter_idx]` 에 `RoleInvocationResult` 를
memoise. replay 시 cache HIT → invoke 생략 → interrupt 가 저장 answer 반환.
terminal (COMPLETED/INCOMPLETE/FAILED) 에서만 entry pop, GraphInterrupt 경로는
유지. LLM schema 에는 `tool_call_id` 비노출 확인 (`tool.tool_call_schema`).

**원인 2 (마지막 출력 첫 줄)** — `_run_agent_streaming` 마지막의
`store.list_all()` 이 **존재하지 않는 API** (v2 schema 전환으로 제거됨) →
AttributeError → CLI 메인 루프의 `except Exception:` 이 silent catch →
`_run_agent_simple(user_input)` 폴백 → **같은 user_input 처음부터 재실행**.
두 번째 agent_loop 의 planner 가 ask_user_question 1-step 뒤 `no_ask_user_handler`
로 조기 종료 → 그 짧은 응답이 "마지막 출력" 으로 보였음.

수정:
- `_run_agent_streaming` 의 `store.list_all()` 섹션 삭제
- CLI 메인 루프의 silent simple-fallback 제거 — `log.exception` + print_error
  로 표면화, 재실행 없음
- `cli/app.py` 에 `structlog.get_logger("cli")` 추가

## 중요 참고사항

### LangGraph interrupt 공식 pitfall
https://langchain-ai.github.io/langgraph/concepts/human_in_the_loop/#code-that-precedes-interrupts-is-replayed

> The part of the node or the specific tool preceding `interrupt` is replayed.
> Any side effects (LLM calls, API calls) that occurred before the interrupt
> will be re-executed when the graph resumes.

공식 해결책 3가지 중 **external cache** 를 선택 (수정 규모 작고 명시적). 장기적으론
subgraph 분리 (공식 권고 2번) 가 더 정통적이나 지금 범위 밖.
